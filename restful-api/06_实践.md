# RESTful API 实践

## 安全规范

### 身份认证与鉴权（Authentication & Authorization）

绝大多数受保护的 API 需要校验用户身份。业内标准的做法是在请求头中使用 `Authorization` 携带 JWT（JSON Web Token）。

```http
// → 请求带上凭证
GET /api/users/profile
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI...

// ← 服务器解析 Token 并返回对应权限的数据
{
	"success": true,
	"data": {"id": 123, "name": "张三", "role": "admin"}
}
```

### 严守输入验证（Input Validation）

永远不要相信客户端传来的数据！必须在服务端进行严格的字段校验。

```json
// ← 当密码过短时，返回 400 状态码及准确的错误提示
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "密码长度至少为 8 位"
  }
}
```

### 强制 HTTPS

生产环境的 API 必须 100% 走 HTTPS 协议，防止中间人抓包窃取 Token 或篡改数据。

- ✓ `https://api.example.com/users`
- ✕ `http://api.example.com/users`

## 性能优化

### 数据分页（Pagination）

严禁直接返回表中的全量数据！列表接口必须支持分页（`page` & `limit`）。

```http
// → 请求第 1 页，限制每页返回 2 条数据
GET /api/users?page=1&limit=2

// ← 响应体除了 data，还必须包含 pagination 对象
{
  "success": true,
  "data": [
    {"id": 1, "name": "张三"},
    {"id": 2, "name": "李四"}
  ],
  "pagination": {
    "currentPage": 1,    // 当前页码
    "itemsPerPage": 2,   // 每页条数
    "totalPages": 50,    // 总页数
    "totalItems": 100    // 总数据量
  }
}
```

### 字段过滤（Sparse Fieldsets）

当客户端只需要部分数据时，允许通过 `fields` 参数指定返回字段，极大减少网络带宽消耗。

```http
// → 只要 id, name 和 email，其余字段不要查
GET /api/users?fields=id,name,email

// ← 响应体得到瘦身
{
	"success": true,
	"data": [
		{"id": 1, "name": "张三", "email": "zhangsan@example.com"}
	]
}
```

### HTTP 缓存策略（Caching）

合理利用 HTTP 协议自带的缓存头，将查询压力挡在服务器之外。

```http
// 策略 A：强缓存（告诉客户端 1 小时内别来烦我，直接用本地的）
Cache-Control: max-age=3600

// 策略 B：协商缓存（客户端拿着数据的指纹 Etag 来问有没有更新）
IF-NOT-Match: "etag-value-12345"
// 服务端比对后若无变化，直接返回 304 Not Modified，不返回任何实体数据
```

## 错误处理机制

### 统一的错误格式

规范的 API 必须定义统一的错误响应结构，包含机器刻度的错误码、人类可读的提示信息以及详细的字段级错误。

```json
{
  "success": false,
  "error": {
    "code": "SPECIFIC_ERROR_CODE",   // 机器可读的业务错误码
    "message": "用户友好的错误信息", // 面向用户的提示信息
    "details": {...},								 // 详细的错误上下文（如字段校验结果，可选）
    "timestamp": "2024-01-15T10:00:00Z"
  }
}
```

### 统一的错误字典

在企业开发中，除了 HTTP 状态码，还应定义一套**业务错误码（通常是全大写字母或特定数字）**，方便前端根据错误码做多语言翻译或特定跳转。

```javascript
// 常见的业务错误码设计
const ERROR_CODES = {
  // 4xx 客户端相关
  'VALIDATION_ERROR': 400,     // 字段验证失败
  'UNAUTHORIZED': 401,         // 未登录或 Token 失效
  'FORBIDDEN': 403,            // 权限不足，禁止访问
  'NOT_FOUND': 404,            // 资源不存在
  'CONFLICT': 409,             // 资源冲突（如手机号已被注册）
  
  // 5xx 服务端相关
  'INTERNAL_ERROR': 500,       // 服务器内部抛出异常
  'SERVICE_UNAVAILABLE': 503   // 服务停机维护中
}
```

## 文档与演进

### 自动化文档（OpenAPI / Swagger）

抛弃手写 Word 接口文档！业内公认的标准是使用 OpenAPI 规范（Swagger）。代码即文档，且支持在线直接测试 API。

```yaml
# swagger.yaml 示例片段
openai: 3.0.0
info:
	title: 用户管理 API
	version: 1.0.0
	description: 提供用户的增删改查功能
paths:
	/api/users:
		get:
			summery: 获取用户列表
			parameters:
				- name: page
					in: query
					description: 页码
					schema:
						type: integer
						default: 1
			responses:
				'200':
					description: 成功
					# ...（省略具体 schema 定义）
```

### 版本控制策略

#### URL 版本控制

在 URL 中明确 API 版本，当业务发送破坏性更新时，保持向后兼容。

```http
// 版本 1
GET /api/v1/users

// 版本 2（可能新增了必填字段，但老客户仍在平稳调用 v1）
GET /api/v2/users
{
	"id": 1,
	"name": "张三",
	"avatar": "https://example.com/avatar.jpg" // 新增字段
}
```

#### 优雅的兼容性处理（Deprecation）

当 API 准备废弃时，老版本接口不能直接删掉，应在返回体重加入警告，给前端留出升级窗口期。

```json
{
  "deprecationWarning": "该 API 版本将在 2024 年 6 月 1 日后停止支持，请升级到 v2",
  "data": {...}
}
```



## 课后实战演练

为了巩固 RESTful API 设计规范，你可以尝试完善以下三个系统的接口设计：

### 图书管理系统

**需求分析**：

- 管理图书信息（标题、作者、ISBN、价格）
- 管理图书分类
- 支持图书的增删改查及按分类筛选

**API 端点设计**：

```http
// 图书相关
GET    /api/books				// 获取图书列表
GET    /api/books/{id}  // 获取特定图书
POST   /api/books       // 创建新图书
PUT    /api/books/{id}  // 更新图书信息
DELETE /api/books/{id}  // 删除图书

// 分类嵌套路由
GET /api/categories 					  // 获取分类列表
GET /api/categories/{id}/books  // 获取某分类下的所有图书
```

**数据结构示例**：

```json
{
  "id": 1,
  "title": "JavaScript 高级程序设计",
  "author": "Nicholas C. Zakas",
  "isbn": "9787115275790",
  "price": 99.00,
  "categoryId": 1,
  "category": { "id": 1, "name": "编程技术" },
  "stock": 50
}
```

### Todo 待办事项

**功能要求**：

创建、查看、更新、删除待办事项；标记完成状态；按状态筛选任务；支持任务优先级。

**数据模型设计**：

```json
{
  "id": 1,
  "title": "学习 RESTful API",
  "description": "完成 API 教程的学习",
  "status": "pending",  // pending, completed, cancelled
  "priority": "high",   // low, medium, high
  "dueDate": "2024-01-20",
  "createAt": "2024-01-15T08:00:00Z"
}
```

**挑战任务**：

- **批量操作**：设计批量更新任务状态的 API。
- **搜索功能**：支持按标题和描述搜索。
- **统计信息**：返回任务完成情况的统计数据接口。

### 社交媒体后端

**功能要求**：

用户注册登录、发布查看动态、关注其他用户、点赞和评论。

**API 结构设计**：

```http
// 认证与用户
POST /api/auth/register  // 用户注册
POST /api/auth/login		 // 用户登录
GET  /api/users/profile  // 获取个人资料

// 动态相关
GET    /api/posts				// 获取动态列表
POST   /api/posts       // 发布新动态
GET    /api/posts/{id}  // 获取特定动态
DELETE /api/posts/{id}  // 删除动态

// 互动相关（巧妙的嵌套设计）
POST /api/posts/{id}/like      // 点赞动态
POST /api/posts/{id}/comments  // 添加评论
GET  /api/posts/{id}/comments  // 获取某条动态的评论列表
```
