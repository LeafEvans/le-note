# RESTful API 进阶

当 API 满足了基本的增删查改（CRUD）后，为了应对复杂的业务场景和高并发的生产环境，我们需要引入更高级的架构设计与安全策略。

## 认证与授权（Authentication & Authorization）

### JWT（JSON Web Token）无状态认证

JWT 就像是一张“数字身份证”，它包含了用户的身份信息，并通过加密签名防止被篡改。它的最大优势是**服务端无状态**（不需要在内存或 Session 中保存登录状态）。

**JWT 结构**：`Header(头部) . Payload(负载) . Signature(签名)`

**登录与签发流程**：

```http
// 客户端提交账号密码
POST /api/auth/login
{
	"email": "user@example.com",
	"password": "password123"
}

// 服务端校验成功，返回 Token
{
	"success": true,
	"data": {
		"token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJpZCI...",
		"user": {
			"id": 123,
			"name": "张三"
		}
	}
}

// 后续请求：客户端在请求头中携带 Token 以证明身份
GET /api/users/profile
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJpZCI...
```

### OAuth 2.0 第三方授权

用于实现“使用 Google / 微信 / GitHub 登录”等功能。

```http
// 引导用户跳往第三方授权页面
GET /api/auth/google/redirect

// 用户同意后，第三方重定向回你的系统，并附带授权码（code）
GET /api/auth/google/callback?code=authorization_code_xxx

// 服务器在后台用 code 换取真实的 Access Token 及用户信息
```

## API 限流与配额（Read Limiting）

为了防止恶意攻击（如 DDos）或单个用户消耗过多服务器资源，必须对 API 进行频率限制。

**标准的限流响应头与状态码**：

```http
// 正常请求时，响应头会告诉客户端剩余的额度
HTTP/1.1 200 OK
X-RateLimit-Limit: 1000        // 基础配额：每小时允许 1000 次请求
X-RateLimit-Limit: 999         // 当前剩余：999 次
X-RateLimit-Reset: 1642694400  // 重置时间：额度恢复的 Unix 时间戳

// 超出限制时，必须返回 429 状态码（Too Many Requests）
HTTP/1.1 429 Too Many Requests
Retry-After: 3600  // 建议客户端等待的时间（秒）

{
	"success": false,
	"error": {
		"code": "RATE_LIMIT_EXCEEDED",
		"message": "请求过于频繁，请稍后再试"
	}
}
```

## 数据缓存策略（Caching）

合理的缓存能大幅降低数据库压力，提升 API 响应速度。缓存通常分为“客户端缓存”与“服务端缓存”。

### 客户端缓存（利用 HTTP 头）

利用 HTTP 协议原生的机制，让浏览器或 CDN 缓存数据。

```http
// 强缓存：直接告诉客户端缓存 1 小时（3600 秒）
GET /api/users/123
Cache-Control: public, max-age=3600
ETag: "a1b2c3d4e5f6"  // 数据的唯一版本指纹

// 协商缓存：客户端再次请求时，带上上次的指纹询问是否过期
GET /api/users/123
IF-None-Match: "a1b2c3d4e5f6"

// 服务端比对后发现数据没变，直接返回 304，不返回任何数据体（极省带宽）
HTTP/1.1 304 Not Modified
```

### 服务端缓存（Redis 旁路缓存模式）

在业务代码中，使用 Redis 作为中间缓存层。经典的 **Cache-Aside** 策略如下：

```javascript
async function getUserProfile(userId) {
  const cacheKey = `user:profile:${userId}`;
  
  // 先检查 Redis 缓存是否命中
  const cacheData = await redis.get(cacheKey);
  if (cacheData) {
    return JSON.parse(cacheData); // 命中缓存，直接返回！
  }
  
  // 缓存未命中，回源查询数据库（耗时操作）
  const user = await database.findUser(userId);
  
  // 将数据库查到的结果写入 Redis，并设置过期时间（如 1 小时）
  if (user) {
    await redis.setex(cacheKey, 3600, JSON.stringify(user));
  }
  
  return user;
}
```

### 微服务架构中的 API 网关（API Gateway）

<img src="../images/restful-api/image-20260320-111805.svg" style="zoom:80%;" />

在微服务架构中，后端会被拆解为无数个小服务。为了不让前端强耦合各个服务的 IP 和端口，通常会引入 **API 网关**作为统一的流量入口。

**API 网关的路由分发示例**：

```json
// 网关就像一个路由器，根据路径特征，将请求转发给内部真实的微服务
{
  "gateway_routes": [
    {
      "path_match": "/api/users/*",      // 匹配用户相关路径
      "target_service": "user-service",  // 转发到用户微服务
      "internal_url": "http://10.0.0.1:3001"
    },
    {
      "path_match": "/api/orders/*",      // 匹配订单相关路径
      "target_service": "order-service",  // 转发到订单微服务
      "internal_url": "http://10.0.0.2:3002"
    }
  ]
}
```

## REST 的局限性与 GraphQL

### RESTful

在高度复杂的前端页面中，REST API 容易产生**请求瀑布流（Under-fetching）或数据冗余（Over-fetching）**。

为了渲染一个“用户详情页”，REST 可能需要发起多次无脑的连环请求：

```http
// 获取基本信息
GET /api/users/123
// 根据获取到的用户 ID，再去拿他的文章列表
GET /api/users/123/posts
// 再根据文章 ID，去拿具体的评论
GET /api/posts/456/comments
```

### GraphQL 

GraphQL 提供了一种全新的数据查询语言。**核心理念：客户端需要什么，就精确索要什么，并且一次请求全部带回。**

```javascript
# 只需要发送一次 POST 请求，按图索骥获取所有关联数据
query {
  user(id: 123) {
    name
    email
    posts {
      title
      comments {
        content
        author {
          name
        }
      }
    }
  }
}
```
